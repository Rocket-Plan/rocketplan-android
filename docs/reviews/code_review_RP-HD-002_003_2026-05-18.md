## Summary

The diff lands the HD-002 try/catch and the HD-003 ordering comment correctly, but it also rewrites the migration lifecycle to make `migrationDeferred` reset-table — work that belongs to the **separate, already-planned** `RP-BUG-026`. That rewrite has a bug: **`migrationDeferred` is never initialized**. Cold-start auth reads for any user with a legacy DataStore token will now always return `null`, re-introducing the spurious sign-out that `RP-BUG-006` was filed to fix.

The test suite added is well-structured and includes the new `clearAll` scenarios, but **none of the new tests assert that Sentry was captured**, which was an explicit requirement of the RP-HD-002 plan.

## P0 — Regression of RP-BUG-006

### What broke

`migrationDeferred` is now declared as a nullable `var`, initialized to `null`:

```kotlin
// SecureStorage.kt:109-110
private val migrationMutex = Mutex()
private var migrationDeferred: Deferred<String?>? = null
```

There are two helpers that *would* start the real migration:

```kotlin
// SecureStorage.kt:112-121 — defined but never called from production code
private fun createMigrationDeferred(scope: CoroutineScope): Deferred<String?> = …
private suspend fun initMigrationDeferred(scope: CoroutineScope) { … }
```

`grep -rn "initMigrationDeferred\|createMigrationDeferred" app/src/main` returns only the definitions themselves. **No production call site invokes either function.** The previous `init { … }` block that eagerly started the migration was removed and not replaced.

`getAuthTokenSync()` then does:

```kotlin
// SecureStorage.kt:145-151
suspend fun getAuthTokenSync(): String? {
    authTokenState.value?.let { return it }
    val deferred = migrationMutex.withLock { migrationDeferred }
        ?: scope.async { null }.also { migrationDeferred = it }   // <-- no-op deferred
    deferred.await()
    return authTokenState.value
}
```

When called with no encrypted token and `migrationDeferred == null`, the elvis branch creates `scope.async { null }` — a deferred that completes with `null` **without ever invoking `migrateLegacyAuthToken()`**. The legacy `readLegacyToken` lambda is never read in production.

### User impact

Anyone whose auth token is still in the legacy DataStore key (the population this whole work item was designed to protect) will get `null` from `getAuthTokenSync()` on cold start and be signed out at the first authenticated request. That is exactly the `RP-BUG-006` symptom.

### Why the unit tests don't catch it

I can't actually rerun the test suite because of the pre-existing compile errors in `OfflineSyncRepositoryTest.kt` and `PusherServiceTest.kt` flagged in the user's report. The test source set as a whole fails to compile, so `testDevStandardDebugUnitTest` aborts before any `SecureStorageTest` case runs.

Independent of that, even if the tests *did* compile and ran, my static read of test `getAuthTokenSync returns migrated legacy token when no encrypted token exists` (lines 87–100) says it should assert `"legacy-token"` but actually receive `null` from the production code shown above. Either the user's reported run was on a different (unsaved) edit, or the tests were not actually rerun against this version of the production file. Either way, **this regression is in the working tree right now** and needs to be fixed before the diff is committed.

### Suggested fix

Restore eager construction of the deferred, but keep the `var` so `clearAll()` can replace it. Two-line change:

```kotlin
private val migrationMutex = Mutex()
private var migrationDeferred: Deferred<String?>? = createMigrationDeferred(scope)
```

…and delete the elvis fallback in `getAuthTokenSync()` (or leave it as belt-and-suspenders, but the primary path should be "the migration started at construction time"). Drop the unused `initMigrationDeferred` helper if no other call site needs it.

This restores `RP-BUG-006`'s fix while keeping the resettability that `clearAll` and the new tests need.

## Scope creep — `RP-BUG-026` was silently bundled

The plan for HD-002/HD-003 was a documentation comment plus a try/catch wrapping `migrateLegacyAuthToken()`. It explicitly says:

> No build.gradle or other-file changes. The catch block is unreachable in current unit tests…

What actually landed:

- New `Mutex` + `var migrationDeferred` lifecycle (resettable).
- `clearAll()` cancels and replaces the deferred.
- `clearAll()` was reordered so the mutex section runs **before** clearing DataStore.
- Three new tests covering `clearAll` ↔ migration race.

That's a faithful implementation of `RP-BUG-026` (*SecureStorage.clearAll() Doesn't Clear migrationDeferred*) — which has its own plan, `plans/plan_rp_bug_026_clearall_migration_deferred_2026-05-18.md`, currently in `state: planned` in the tracker.

**Recommendation:** decide explicitly whether this PR is a bundle (HD-002 + HD-003 + BUG-026) or scoped only to the HDs. If bundled, update `BUG_TRACKER.md` to mark `RP-BUG-026` as `fixed` too, and reference its plan from the commit message. If scoped, revert the lifecycle / `clearAll` changes and the three `clearAll`-race tests to a separate diff.

I'd recommend the bundle — splitting now creates merge conflicts in the same lock-protected section.

## Test review

The new tests are well-structured (`StandardTestDispatcher`, `CompletableDeferred` to gate timing, `runTest` with virtual-time). But two material gaps:

### T1 — No assertion that Sentry is captured for HD-002

The RP-HD-002 plan's test plan lists:

> Stub `legacyTokenReader = { error("boom") }`. Mock `Sentry.captureException` via `mockkStatic(Sentry::class)`. Assert `getAuthTokenSync()` returns `null`. Verify `Sentry.captureException` called exactly once… Verify the scope was tagged with `event=auth_migration_failed`.

The user's summary says these two tests were added, but neither appears in the file as written. The cases covered are:
- 1×: migrated-legacy happy path
- 1×: no-legacy, no-encrypted
- 1×: save-vs-migrate race (BUG-006 from earlier)
- 1×: timeout (BUG-006 from earlier)
- 3×: clearAll race scenarios (BUG-026)
- 1×: no-legacy fast no-op (redundant with test 2)

Zero tests exercise the `catch (e: Throwable)` branch added in HD-002. Without a Sentry verification, the most important behavior of HD-002 — *the failure is broken out of silence* — is uncovered.

**Action:** add the two cases the plan specified. A `mockkStatic(Sentry::class)` block with a `verify(exactly = 1) { Sentry.captureException(any()) }` and a scope-tag assertion is enough.

### T2 — Test suite cannot run today

`PusherServiceTest.kt` and `OfflineSyncRepositoryTest.kt` have pre-existing compile errors. While these aren't this diff's responsibility, the practical effect is that **no green/red signal exists for the SecureStorage work either**. Recommend filing a separate ticket to either fix or exclude the broken tests so the unit-test gate is meaningful again. Without that, every future change to `SecureStorage` is shipping on visual inspection.

## Findings (lower-priority)

### F1 — `clearAll()` now uses `apply()` on EncryptedSharedPreferences

The change at `SecureStorage.kt:343` removed the comment but kept `encryptedPrefs.edit().clear().apply()`. **Note that `RP-BUG-005` is already filed for this exact pattern** (non-blocking `apply()` write to encrypted prefs for auth tokens). Out of scope for this diff, but flagging so it's not "forgotten as fixed" — `RP-BUG-005` remains `open` and the same concern applies inside `clearAll`.

### F2 — `@OptIn(ExperimentalCoroutinesApi::class)` annotation

Added at the class level. None of the APIs touched here (`async`, `Deferred`, `Mutex`, `withLock`, `withTimeoutOrNull`) are actually `@ExperimentalCoroutinesApi`. The annotation is harmless but misleading. **Suggested action:** delete it, or move it to whichever specific call needs it (none, on my read).

### F3 — `legacyTokenReader` is dead code in production

Even with the fix in P0 above, the `legacyTokenReader` / `legacyTokenClearer` constructor parameters were added for test injection only. That's fine, but a one-line comment above the secondary constructor explaining "test-only injection points" would prevent a future reader from wondering why a "secure storage" class accepts arbitrary lambdas in its primary constructor. (This is HD-003-shaped guidance: the *order* is documented, the *injection points* should be too.)
