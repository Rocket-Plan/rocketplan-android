# Fix Plan: Critical P0 Bundle (RP-BUG-001, RP-BUG-006)

**Bug ID(s):** RP-BUG-001, RP-BUG-006
**Author:** jeremie@rocketplantech.com
**Date:** 2026-05-13
**State:** shipped
**Review:** [code_review_RP-BUG-001_006_2026-05-18.md](../reviews/code_review_RP-BUG-001_006_2026-05-18.md)

---

## Summary

Bundled fix for the two drop-everything P0s:

1. **RP-BUG-001** — `OfflineDatabase` falls back to destructive migration whenever `BuildConfig.DEBUG` is true. The guard is fragile (any prod build that ships with `DEBUG=true`, or any future flavor that inherits debuggable, silently wipes offline data on a missing migration). Replace with an explicit, flavor-scoped opt-in plus a non-destructive failure path that reports to Sentry instead of nuking the DB.
2. **RP-BUG-006** — `SecureStorage` launches `migrateLegacyAuthToken()` from `init` while `getAuthTokenSync()` reads `authTokenState.value` without coordination. Cold-start callers can observe `null` and trigger a spurious 401 / sign-out before migration completes. Gate `getAuthTokenSync()` on a `Deferred<String?>` (or `Mutex`) so the first read awaits migration completion.

Both are pre-existing latent bugs found by review; neither has a Sentry signal today, so step 1 of the rollout is "add observability so we can tell if the fix worked."

## Order of work

Do **RP-BUG-006 first** (it fires on every cold start of any user who still has a legacy DataStore token) then **RP-BUG-001** (latent until the next schema bump — buys us margin but must land before the next migration).

Land both in a single release.

## Implementation Notes

### Step 1 — RP-BUG-006: gate `getAuthTokenSync()` on migration completion

`SecureStorage.kt`:

```kotlin
private val migrationDeferred: Deferred<String?> = scope.async {
    // Bound the migration so a slow/stuck DataStore can't hang every
    // auth read forever. Falling back to the encrypted-prefs value is
    // strictly better than blocking the request path.
    withTimeoutOrNull(MIGRATION_TIMEOUT_MS) { migrateLegacyAuthToken() }
}

suspend fun getAuthTokenSync(): String? {
    // Fast path: migration already finished and populated the flow.
    authTokenState.value?.let { return it }
    // Slow path: wait for the one-shot migration. After it completes,
    // authTokenState is authoritative.
    migrationDeferred.await()
    return authTokenState.value
}
```

Notes:
- Replace the `init { scope.launch { migrateLegacyAuthToken() } }` block with the `async` above (one-shot, idempotent).
- `migrateLegacyAuthToken()` already updates `authTokenState.value` on success; keep that as the single write site so the `Flow` consumers (`getAuthToken()`) still see the migrated value.
- **Guard against clobbering a fresh save:** change the in-migrator write from `authTokenState.value = legacyToken` to `authTokenState.compareAndSet(null, legacyToken)`. If a user signs in fresh while migration is still in flight, the save wins and the legacy token is not restored over the new one.
- Add a debug log around migration start/end (no remote log unless we later see Sentry signal).
- No-op when no legacy token exists — `migrationDeferred.await()` returns `null` quickly and the flow value remains whatever was loaded from `encryptedPrefs` at construction time.
- `MIGRATION_TIMEOUT_MS = 5_000L` — generous upper bound on a DataStore read. Past that, fall back to the encrypted-prefs value.
- **Scope of the fix:** `getAuthTokenSync()` is the only accessor gated on migration completion. Flow subscribers via `getAuthToken()` retain normal StateFlow behavior and may observe a transient `null → <token>` during the migration window — that's correct for reactive consumers and isn't the path that produced the spurious sign-outs.

### Step 2 — RP-BUG-001: explicit opt-in + safe failure

`app/build.gradle.kts` — set on the existing **environment** flavors only (the project uses two flavor dimensions, `environment` × `device`; combined variant names like `devStandard` are synthesized and are not where fields get configured):

```kotlin
productFlavors {
    create("dev")     { /* existing fields… */ buildConfigField("boolean", "ALLOW_DESTRUCTIVE_MIGRATION", "true")  }
    create("staging") { /* existing fields… */ buildConfigField("boolean", "ALLOW_DESTRUCTIVE_MIGRATION", "false") }
    create("prod")    { /* existing fields… */ buildConfigField("boolean", "ALLOW_DESTRUCTIVE_MIGRATION", "false") }
    // No changes to the `flir` / `standard` device-dimension flavors.
}
```

`OfflineDatabase.kt` `buildDatabase(...)`:

```kotlin
Room.databaseBuilder(context, OfflineDatabase::class.java, DATABASE_NAME)
    .addMigrations(/* … existing list … */)
    .apply {
        if (BuildConfig.ALLOW_DESTRUCTIVE_MIGRATION) {
            fallbackToDestructiveMigration()
        }
        // No else: in staging/prod, Room will throw IllegalStateException
        // on a missing migration. Catch in getInstance() and report.
    }
    .build()
```

Wrap `getInstance` to capture-and-rethrow (do not swallow):

```kotlin
fun getInstance(context: Context): OfflineDatabase =
    instance ?: synchronized(this) {
        instance ?: try {
            buildDatabase(context).also { instance = it }
        } catch (e: IllegalStateException) {
            // Use withScope to match the SDK shape already in use elsewhere
            // (MainActivity.kt:589, ProjectsFragment.kt:312 use plain
            // Sentry.captureException — the trailing-lambda configurator
            // form isn't proven on this SDK version).
            Sentry.withScope { scope ->
                scope.setTag("event", "missing_room_migration")
                Sentry.captureException(e)
            }
            throw e
        }
    }
```

Rationale: a crash on missing migration in prod is **strictly better** than silent data wipe — it surfaces the deploy mistake before any user loses data and gives us a Sentry signal.

**Operational follow-up:** once this lands, the next Room schema bump that ships without a registered migration will hard-crash production on first launch. That's the intended behavior, but flag it in the next schema-bump PR description and validate the migration against a prod-shape DB before tagging the release.

## Observability

- **RP-BUG-006:** keep the new `SecureStorage` debug logs (`legacy auth token migration: start|no legacy token|migrated`) local-only under the existing `SecureStorage` tag so cold-start migration can be confirmed in QA without adding production noise.
- **RP-BUG-001:** capture the missing-migration crash path to Sentry with `event=missing_room_migration`, then rethrow. This deliberately produces one actionable production/staging signal per bad upgrade path instead of silently wiping offline data.
- **Noise control:** no repetitive remote progress logging is added; only the terminal missing-migration failure is sent remotely.
- **QA verification signal:** the auth-race regression is visible in `adb logcat` by tag, and the missing-migration path is visible as a single Sentry-tagged event in non-dev builds.

## Test Plan

- [ ] Unit test: `SecureStorageTest` — pre-seed legacy DataStore token, then immediately call `getAuthTokenSync()` from many coroutines; all return the migrated token, none return null.
- [ ] Unit test: same harness with no legacy token; `getAuthTokenSync()` returns null and migration completes without writing.
- [ ] Unit test: save-vs-migrate race — pre-seed legacy token, in the same tick call `saveAuthToken("fresh")` and `getAuthTokenSync()`; final state is `"fresh"`, not the legacy value (validates `compareAndSet` guard).
- [ ] Build matrix: confirm `BuildConfig.ALLOW_DESTRUCTIVE_MIGRATION` is `false` for any `staging*` and `prod*` variant, `true` for any `dev*` variant. Spot-check `stagingStandardDebug`, `prodStandardRelease`, `devStandardDebug`, `devFlirDebug`.
- [ ] Manual QA — auth race (tablet `30407ef`):
  1. Prereq: install a build where the user still has a legacy DataStore token. Cold start. Watch `logcat` for migration logs and any 401 sign-out.
  2. Expected: first authenticated request succeeds, no sign-out, migration completes once.
- [ ] Manual QA — missing migration (dev flavor):
  1. Prereq: bump schema version locally without writing a migration in a `devStandardDebug` build.
  2. Expected: DB wipes (allowed in dev). Repeat in `stagingStandardDebug`: app crashes with `IllegalStateException`, Sentry receives a `missing_room_migration` event.

## Rollback Plan

Both changes are localized and independent at runtime:

- RP-BUG-006: revert `SecureStorage.kt` to the `init { launch }` form. No data-format change, safe at any point.
- RP-BUG-001: revert `OfflineDatabase.kt` + `build.gradle.kts` buildConfigField. No persisted state depends on the flag.

If a hotfix is needed mid-rollout, the two patches can be reverted independently.

## Dependencies

- Requires: none (no server change).
- Blocking: must land **before** the next Room schema bump (otherwise prod crashes on the missing migration we'd just added a guard for).
